# tlive 飞书 IM 单聊增强规划摘要

更新时间：2026-04-08

## 这份文档的作用

这是一份给后续 agent 快速接手用的摘要文档。

它只回答三件事：

- 目标是什么
- 现状到哪一步了
- 接下来优先做什么

详细交互方案请看：

- [feishu-im-interaction-proposal-cn.md](/home/glwuy/workspace/tlive/docs/feishu-im-interaction-proposal-cn.md)

## 产品目标

飞书单聊的目标不是做“命令大全”，而是把 `tlive` 做成：

`Claude Code 的远程工作入口`

核心闭环：

- 发任务
- 看进度
- 做决策
- 收结果
- 进入下一步

设计原则：

- 默认优先点选，不要求用户记命令
- 当前状态要清楚
- 用户需要介入时要立刻可见
- 中间过程可感知，并且可打断

## 当前代码基础

当前已有的能力足够支撑下一步产品化，不是从零开始。

已具备：

- 飞书单聊消息接入
- 长连接接收事件
- 卡片按钮回调 `card.action.trigger`
- 文本、图片、文件、引用回复输入
- 权限审批卡片
- 问题选择卡片
- 文本 fallback
- 会话切换、目录切换、停止执行等控制能力
- 过程刷新和最终结果回传

当前缺的不是底层能力，而是交互组织。

## 当前问题

主要问题有这几个：

- 用户仍然需要记不少命令
- 中间过程感知不够强
- 等权限、等回答、已完成这些状态没有统一产品表达
- 会话恢复还是偏工程视角
- 任务完成后缺少自然的下一步入口

## 规划结论

飞书单聊后续建议采用这套交互结构：

- 机器人菜单：全局入口
- 首页卡：进入单聊后的起点
- 任务状态卡：当前任务主容器
- 折叠过程区：展示过程摘要
- 跟随气泡：承接轻量下一步
- 自然语言：承接复杂表达

一句话：

- 菜单做导航
- 卡片做状态和决策
- 跟随气泡做下一步
- 输入框做复杂任务

## 优先级

### P0

- 首页卡
- 执行中状态卡
- 等待权限卡
- 完成卡
- 可打断入口显式化

### P1

- 机器人菜单
- 会话恢复卡
- 跟随气泡下一步
- 失败恢复卡

### P2

- 更细粒度的过程摘要
- 更强的附件理解入口
- 更细的权限分层

## 关于过程展示

不建议把完整思考过程直接摊开。

建议做法：

- 首屏只展示状态摘要
- 用折叠面板展示“工作日志摘要”
- 用启发式中间播报让用户感知进展

关键原则：

- 让用户知道系统还在工作
- 让用户知道什么时候该介入
- 让用户可以随时打断
