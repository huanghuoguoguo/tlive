# tlive 用户体验改版计划（中文）

更新时间：2026-04-08

## 背景

当前 `tlive` 已具备多平台接入能力，但从首次接触产品的用户视角看，入口信息仍然偏“能力展示”和“技术结构说明”。

如果产品目标是让个人用户尽快在手机上远程使用 Claude Code，那么中文语境下更合适的默认路径应当收敛为：

1. 安装 `tlive`
2. 运行 `/tlive setup` 或 `tlive setup`
3. 默认优先选择飞书
4. 完成一次最小配置
5. 在飞书中发出第一条消息并收到回复

这份文档用于记录后续改版方向，便于跨 session 持续推进。

## 核心结论

- 中文文档应明确把“个人用户推荐飞书”作为默认路径
- 文档入口、平台推荐顺序、setup 引导顺序需要一致
- 首次使用阶段应弱化技术概念，强化上手路径和成功反馈
- 现阶段优先改文档和 setup wizard，不调整 bridge 核心运行逻辑

## 改版目标

### 目标一：让用户更快开始

用户在首页或入门文档中，应当在 30 秒内知道三件事：

- `tlive` 是什么
- 我现在该选哪个平台
- 我完成配置后会得到什么

### 目标二：收敛默认路径

中文文档中不再把 Telegram、Discord、飞书、QQ Bot 作为完全并列的首选项，而是明确：

- 飞书：个人用户默认推荐
- Telegram：备选
- Discord：团队场景备选
- QQ Bot：补充选项

### 目标三：降低首次使用的理解成本

在用户真正跑通第一条消息前，不应该要求其优先理解：

- Bridge / Core / Hook 的内部架构
- 所有平台差异
- 全量环境变量
- 所有命令开关

## 本轮建议范围

本轮只做以下改动：

- 调整中文文档入口
- 调整中文平台推荐顺序
- 调整飞书文档定位
- 调整 setup wizard 中的平台展示顺序和提示语

本轮不做以下改动：

- 不改 bridge 核心运行逻辑
- 不改 Feishu adapter 行为
- 不改权限审批协议
- 不改英文文档的整体平台推荐策略

## 文件级改动计划

### 1. README_CN.md

目标：把中文首页从“能力总览”改成“默认上手入口”。

建议修改：

- 在标题区明确加入“个人用户推荐飞书”
- 快速开始改成单一路径：安装 -> `/tlive setup` -> 选择飞书 -> 启动 -> 发第一条消息
- 把“选择平台（Telegram/Discord/飞书/QQ Bot）”改成“默认推荐飞书，其他平台可选”
- 首屏保留用户最关心的结果：
  - 手机发任务
  - 手机收进度
  - 手机点权限审批
- 架构图、Bridge/Core 说明后移

预期效果：

- 用户先理解如何开始，而不是先理解系统实现

### 2. docs/getting-started-cn.md

目标：把入门文档变成真正的黄金路径。

建议修改：

- 在前置条件之后直接说明：
  - 中文环境个人用户默认推荐飞书
  - 如果你已经长期使用 Telegram，再选 Telegram
- 平台对比表改成“默认推荐 + 备选方案”结构，而不是并列比较
- `方案 A：AI 引导配置` 保持为第一推荐路径
- 示例配置默认改成飞书，而不是 Telegram
- “试一试”部分优先展示飞书 IM Bridge，而不是先强调 Web 终端
- 增加明确的成功判定文案：
  - 你已经可以在飞书中给机器人发消息
  - 你会在 Claude 需要权限时收到审批卡片

预期效果：

- 用户看完文档后，不需要自己做平台决策

### 3. docs/setup-feishu-cn.md

目标：把飞书文档从“一个可选平台说明”提升为“中文个人用户默认教程”。

建议修改：

- 开头直接注明：这是中文个人用户的默认推荐方案
- 降低心理门槛，强调：
  - 不需要公网 IP
  - 不需要自己部署 webhook
  - 使用长连接即可
- 将“飞书配置步骤较多”的表述收敛，避免第一印象是复杂
- 第六步配置 tlive 时，优先展示 `/tlive setup`
- 第七步验证部分改写得更结果导向，强调“出现机器人回复/审批卡片即成功”

预期效果：

- 用户更愿意直接选择飞书，而不是因“看起来麻烦”提前流失

### 4. bridge/src/setup-wizard.ts

目标：让 CLI 交互顺序和中文文档保持一致。

当前问题：

- 平台顺序是 `Telegram`, `Discord`, `Feishu`, `QQBot`
- 这与“中文个人用户优先飞书”的定位冲突

建议修改：

- 平台顺序调整为 `Feishu`, `Telegram`, `Discord`, `QQBot`
- 在问题文案里加入简短提示，例如：
  - `Feishu (recommended for personal users)`
- 如果后续愿意继续优化，可再考虑：
  - 为中文环境默认预选飞书
  - 在选择平台前加一句说明，而不是只列选项

预期效果：

- 用户在命令行中看到的默认导向与文档一致

## 信息架构建议

中文文档首页和入门文档建议统一成以下顺序：

1. 你能用它做什么
2. 个人用户推荐飞书
3. 最快上手路径
4. 配置成功后会看到什么
5. 其他平台和高级能力
6. 架构说明和配置细节

不建议继续采用当前这种偏开发者视角的顺序：

1. 先列能力和平台
2. 再讲架构
3. 最后让用户自己决定怎么上手

## 文案原则

后续修改时，建议统一遵循以下原则：

- 默认给结论，不把平台选择压力交给用户
- 先讲结果，再讲原理
- 先讲最短路径，再讲高级能力
- 先讲成功判定，再讲故障排查
- 中文文档优先围绕中文个人用户的真实使用环境设计

## 实施顺序

建议按以下顺序推进：

1. 修改 `README_CN.md`
2. 修改 `docs/getting-started-cn.md`
3. 修改 `docs/setup-feishu-cn.md`
4. 修改 `bridge/src/setup-wizard.ts`
5. 完成后统一检查文档与 CLI 引导是否一致

## 验收标准

如果本轮改版完成，应至少满足以下结果：

- 中文用户第一次看文档时，能明确知道默认推荐飞书
- 中文用户不需要自己比较多个平台后再决定
- `README_CN.md`、`getting-started-cn.md`、`setup-wizard.ts` 的默认导向一致
- 用户可以顺着同一条路径完成首次配置和首次验证

## 后续可选扩展

如果这一轮效果确认，可以继续考虑：

- 为中文首页增加“飞书快速开始”单独区块
- 为 setup wizard 增加更明确的推荐说明
- 为飞书配置文档补截图
- 为首次成功配置增加更显式的“下一步建议”
- 视受众再决定是否同步英文文档的信息架构

## 架构现状评审

更新时间：2026-04-08

这一节用于回答以下问题：

- 当前代码架构是否已经处于最佳状态
- 当前架构是否符合现阶段需求
- 是否存在不必要的冗余抽象
- 代码规范是否足够稳定和可持续

结论先行：

- 现状不是“最佳状态”，但整体仍然是可用且方向正确的
- 对当前“本地单用户 + 多 IM 平台 + Claude Code 桥接”的需求，整体是匹配的
- 真正的问题不是抽象太多，而是编排层过胖、平台分支分散、依赖注入边界不够清晰
- 代码规范处于中上水平，但自动化约束不足

## 当前架构的优点

### 1. 总体分层方向是对的

当前 Bridge 侧已经有相对明确的结构分工：

- `channels/` 负责平台适配
- `engine/` 负责会话、命令、权限、渲染等编排逻辑
- `providers/` 负责 Claude SDK 集成
- `store/` 负责 binding、去重和锁
- `config.ts` 负责配置装载和校验

这说明项目不是“所有逻辑堆在入口文件”，基础架构方向没有走偏。

### 2. 适配器抽象是有价值的

`BaseChannelAdapter` 和动态按需加载不是冗余抽象，而是当前需求下必要的基础设施。

保留价值：

- 不同 IM 平台的生命周期和发送能力可以统一接入
- 多平台支持不会直接污染主入口
- 动态加载未启用的平台可以减少依赖和内存占用

结论：

- `channels/` 这一层应保留
- 不建议因为“看起来抽象”而把平台逻辑重新塞回主流程

### 3. TypeScript 严格模式和测试覆盖是明显优点

当前已经具备：

- `strict: true`
- 独立测试目录
- 针对 adapter、config、renderer、bridge-manager 等模块的测试

这说明代码质量并不是完全靠手工兜底。

## 当前架构的主要问题

### 1. BridgeManager 已经明显过胖

`BridgeManager` 当前承担了过多职责，包括但不限于：

- adapter 注册与生命周期管理
- hook 通知分发
- SDK session 状态协调
- 文本前置解析与 reply routing
- 清理定时器管理
- 平台特例处理

不过，这一问题已经开始被逐步拆解。当前已从 `BridgeManager` 中抽出的模块包括：

- `session-state.ts`
- `permission-coordinator.ts`
- `command-presenter.ts`
- `callback-dispatcher.ts`
- `ingress-coordinator.ts`
- `message-loop-coordinator.ts`
- `text-dispatcher.ts`
- `query-orchestrator.ts`

这说明当前问题已经从“所有职责都堆在一个类里”，收敛为“主编排类仍偏大，但边界正在变清晰”。

这类模块在早期迭代阶段很常见，但继续扩展后会带来几个问题：

- 改一个功能容易波及其他逻辑
- 测试定位困难
- 新功能倾向于继续堆到这个类里
- 维护者需要长期记住过多上下文

判断：

- 这是当前最需要控制复杂度的地方

### 2. 平台抽象只完成了一半

虽然有 `BaseChannelAdapter`，但展示层和交互层仍有不少平台分支散落在：

- `CommandRouter`
- `BridgeManager`
- 部分消息发送逻辑

也就是说，当前抽象更像是：

- 传输能力被抽象了
- 但“如何表达同一个业务动作”还没有统一模型

这会导致两个问题：

- 新增命令时要在多个地方补平台分支
- 平台差异会逐渐从 adapter 层渗透到 engine 层

判断：

- 不是抽象太多，而是抽象没有完全落到业务呈现层

### 3. 依赖注入边界不够清晰

当前 `context.ts` 使用 `globalThis` 保存运行时上下文，多个模块通过 `getBridgeContext()` 隐式读取：

- `store`
- `llm`
- `core`
- `defaultWorkdir`

这种方式的优点是写起来快，但缺点也明显：

- 依赖关系不够显式
- 模块复用困难
- 单测替换依赖时不够自然
- 后续重构时容易出现隐式耦合

判断：

- 对当前单进程场景仍可接受
- 但从长期可维护性看，不是理想状态

### 4. 存储和锁模型适合当前需求，但不是通用解

`JsonFileStore` 当前采用：

- 同步文件读写
- 进程内内存锁
- 本地 JSON 持久化

这对当前需求是合理的，因为当前更像：

- 单机运行
- 单用户使用
- 本地守护进程

但它不适合被误认为“通用架构”：

- 多进程下锁不成立
- 进程异常退出后锁状态不保留
- 更复杂的后台服务能力会受到限制

判断：

- 当前不是 bug
- 但后续若向更强 daemon 模式发展，需要重新设计

### 5. 工程规范还不够自动化

目前的优势是：

- 类型约束存在
- 模块命名整体可读
- 测试覆盖面尚可

不足是：

- 虽然已经补了 `Biome`、`.editorconfig` 和统一的 `lint/check` 脚本，但分层约束还没有完全制度化
- 代码风格约束已经初步工程化，但模块边界仍主要依赖维护者判断
- 一些核心文件虽然在持续拆分，但主编排链仍然偏长

判断：

- 这不再是“缺少基础工具链”，而是“工程化基础已经补上，但架构约束还需要继续收口”

## 是否符合当前需求

如果当前需求定义为：

- 本地运行
- 个人或小团队使用
- 主要做手机远程桥接
- 多平台但仍以少数平台为主
- Claude Code 是核心运行对象

那么当前架构总体上是符合需求的。

更准确地说：

- 不是过度设计到完全脱离现实
- 也不是简单到马上要重写
- 而是已经进入“可用，但需要主动控复杂度”的阶段

## 是否存在不必要的冗余抽象

结论：有少量，但不是主要矛盾。

### 应保留的抽象

- `BaseChannelAdapter`
- `providers/`
- `store/` 接口层
- `config.ts` 的集中配置与校验
- `ConversationEngine` 这种相对单一职责的模块

这些抽象和当前需求是匹配的，不建议为了“精简”而撤掉。

### 需要警惕的地方

不是“抽象过多”，而是：

- 有些抽象没有把复杂度真正封装住
- 最终还是回流到了 `BridgeManager` 和 `CommandRouter`

所以真正应该做的是：

- 补齐抽象边界
- 减少 engine 层中的平台特例
- 把大类拆瘦

而不是简单删除抽象层。

## 代码规范评价

综合评价：

- 类型安全：较好
- 模块命名：较清晰
- 测试覆盖：较好
- 风格一致性：中等
- 自动化规范：偏弱

可以认为当前代码规范处于：

- 已经比“脚本堆积型项目”好不少
- 但距离“长期稳定协作的工程化仓库”还有一段距离

## 优先改造建议

### 优先级 1：先拆 BridgeManager

建议拆分方向：

- `IngressCoordinator` / `MessageLoopCoordinator`
- `TextDispatcher`
- HookDispatcher / PermissionDispatcher
- SessionOrchestrator / QueryPipeline

其中前 3 项和 query 执行器已经部分落地，下一步更值得继续拆的是：

- hook 通知和本地 session 输入路由
- Claude 查询执行链与 renderer 生命周期
- 更显式的依赖注入边界

目标不是一次性大重构，而是持续把“会继续长胖的职责”抽出去。

### 优先级 2：把平台展示逻辑从 CommandRouter 中抽离

建议引入一层轻量 presenter / responder，例如：

- `command-presenters/`
- `platform-response-builder/`

让命令处理只负责：

- 解析命令
- 执行业务动作
- 产出统一语义结果

再由平台展示层负责：

- Telegram HTML
- Discord embed
- Feishu 卡片/文本

### 优先级 3：逐步弱化 global context

不建议一次性全改，但建议新代码尽量避免继续扩大 `getBridgeContext()` 的使用范围。

更理想的方向：

- 显式构造依赖
- 通过构造函数注入
- 在入口层统一组装

### 优先级 4：补齐 lint / formatting 约束

这一项已经完成最小落地：

- 已增加 `lint` / `check` 脚本
- 已使用 `Biome` 作为统一规则来源
- 已补仓库级 `.editorconfig`

后续更值得做的是：

- 在 CI 中持续约束这些检查
- 为大文件拆分建立更明确的演进规则
- 继续减少“靠人工记忆维持”的边界

## 推荐策略

不建议现在做“大重写”。

更合适的路线是：

1. 先完成面向用户的入口文档和 setup 路径优化
2. 在不改行为的前提下，逐步拆瘦 `BridgeManager`
3. 把命令响应的展示层平台分支收口
4. 在工具链已补齐的基础上，继续把编排链条拆成更清晰的 coordinator

换句话说：

- 用户体验问题优先修正入口
- 架构问题优先控制复杂度增长
- 不做高风险、低收益的整体重构
