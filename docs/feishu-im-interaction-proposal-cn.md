# tlive 飞书单聊交互增强建议

更新时间：2026-04-08

## 目标

这份文档不是讨论底层实现，而是给出一版更利好用户使用的飞书单聊交互方案。

核心目标只有三个：

- 少记命令
- 少做无效判断
- 更快进入下一步

也就是说，飞书单聊不应该主要像一个“命令行远程壳”，而应该更像一个：

`可点选的远程工作台 + 可补充自然语言的控制入口`

## 交互原则

后续所有飞书单聊设计建议遵循这几条：

- 全局高频动作放到常驻入口
- 当前任务动作放到任务卡片
- 轻量下一步放到消息下方
- 复杂任务仍然允许自然语言输入
- 命令保留给高级用户，但不作为默认主路径

## 建议的交互层次

建议把飞书单聊拆成四层交互。

### 第一层：机器人菜单

这是单聊里的全局导航，不依赖当前任务上下文。

适合放：

- 新任务
- 继续上次任务
- 当前状态
- 会话列表
- 停止执行
- 使用帮助

设计要求：

- 菜单项不要太多，控制在 4 到 6 个高频动作
- 文案尽量是动作，不是概念
- 不要把低频设置项塞进一级入口

推荐菜单结构：

- 开始工作
- 继续上次
- 当前状态
- 会话列表
- 停止执行
- 帮助

产品价值：

- 用户不需要记 `/new` `/sessions` `/stop`
- 菜单承担“我现在能做什么”的入口心智

## 第二层：首页卡

这是用户进入单聊后最适合看到的一张卡。

首页卡的目标不是展示所有信息，而是帮助用户最快开始。

建议包含：

- 当前是否有正在执行的任务
- 当前绑定的项目目录
- 最近一次任务摘要
- 3 到 5 个主操作按钮

推荐按钮：

- 新任务
- 继续上次任务
- 查看当前状态
- 查看最近会话
- 使用帮助

如果当前已有执行中的任务，则首页卡可自动切换为“当前任务概览卡”。

产品价值：

- 用户打开单聊后不用思考第一步该输入什么
- 比欢迎语和命令列表更适合长期使用

## 第三层：任务状态卡

这是最核心的一层。

每个任务都应该尽量映射成一张状态卡，而不是让用户自己从长文本里理解状态。

建议统一这几种状态：

- 执行中
- 等待权限
- 等待用户回答
- 已完成
- 已失败
- 已中断

### 状态卡的结构建议

为了兼顾“首屏清晰”和“过程可追踪”，建议任务状态卡统一采用两层结构：

- 首屏摘要区
- 可展开的过程区

首屏摘要区只放用户最关心的信息：

- 当前状态
- 当前任务摘要
- 最近一步动作
- 是否需要用户介入
- 主操作按钮

过程区默认折叠，用户需要时再展开。

适合放在过程区里的内容：

- 最近 3 到 5 条执行摘要
- 最近读取/修改的文件
- 最近一次工具动作
- 最近一次失败原因
- 下一步准备做什么

这里建议优先使用飞书卡片的 `折叠面板（collapsible_panel）` 承载次要信息，而不是把所有执行过程直接摊在卡片首屏。

设计原则：

- 首屏解决“现在发生了什么”
- 折叠区解决“它刚才做了什么”

不建议在折叠区直接展示未经处理的大段思考文本。更合适的是展示“工作日志摘要”。

### 执行中卡

建议展示：

- 用户原始任务摘要
- 当前阶段
- 最近一次工具动作
- 已运行时长
- 可执行的下一步按钮

按钮建议：

- 停止执行
- 查看摘要
- 显示详细过程

进一步建议：

- 首屏只展示当前阶段和最近一步
- 折叠区展示最近若干条过程摘要
- 如果用户长时间未介入，也能靠首屏知道任务还活着

### 等待权限卡

建议展示：

- 当前要执行的动作
- 风险等级
- 影响对象
- 推荐动作

按钮建议：

- 允许一次
- 拒绝
- 本次会话内持续允许

如果是 Bash，建议额外展示：

- 命令摘要
- 是否涉及写文件
- 是否涉及联网
- 是否可能破坏性

### 等待回答卡

适合 AskUserQuestion 或需要用户补充信息的场景。

建议展示：

- 问题本身
- 选项
- 为什么需要用户回答

如果是单选：

- 直接按钮选择

如果是多选：

- 按钮勾选 + Submit

如果是开放文本：

- 卡片提示“也可以直接回复文字”

### 完成卡

完成后不要只停在一长段文本。

建议展示：

- 做了什么
- 改了哪些文件
- 是否跑了测试
- 当前结论
- 风险或待确认项

按钮建议：

- 查看改动
- 解释过程
- 运行测试
- 继续修复
- 新任务

完成卡的折叠区建议展示：

- 本次关键步骤摘要
- 主要改动文件
- 测试和验证情况
- 未完成项

### 失败卡

失败时最重要的是帮助用户恢复。

建议展示：

- 哪一步失败
- 失败原因摘要
- 是否可重试

按钮建议：

- 重试
- 提高 effort
- 换模型
- 解释失败原因
- 从当前目录重新开始

失败卡的折叠区建议展示：

- 最后一个阶段
- 最近一步工具动作
- 错误摘要
- 建议恢复路径

## 第四层：跟随气泡

跟随气泡很适合做“轻量下一步”。

它不适合承接复杂状态，但很适合承接低成本选择。

适合场景：

- 任务完成后给出 2 到 3 个下一步
- 用户进入单聊后给出默认建议
- 模型输出后给出推荐追问

推荐用法：

- 继续这个任务
- 解释一下
- 运行测试

产品价值：

- 比再发一整张卡更轻
- 比让用户自己输入更省操作

跟随气泡尤其适合承接“任务完成后”的轻量下一步，而不是承接复杂状态本身。

推荐放在这些场景：

- 完成后：继续这个任务 / 解释一下 / 运行测试
- 失败后：重试 / 看原因 / 换模型
- 等待回答后：补充说明 / 使用默认方案

## 过程可感知，而不是只在结束时才发消息

当前单聊体验如果只在任务做完后发最终消息，会有两个问题：

- 用户在执行过程中没有感知，不知道系统是否还在工作
- 用户很难在中途及时打断或修正方向

因此，后续交互建议明确采用：

`启发式中间播报 + 最终结果收敛`

而不是：

`全程沉默 -> 一次性发最终结果`

### 中间播报原则

中间播报不是实时转播所有过程，而是只在“用户该知道的时候”告诉用户。

建议遵循这几个原则：

- 有明显阶段变化时播报
- 长时间无感知输出时播报
- 需要用户介入时立即播报
- 风险或失败出现时立即播报
- 避免把所有细粒度动作都推给用户

### 建议的播报触发时机

推荐在以下时机发中间过程消息或更新状态卡：

- 任务刚开始
- 从“读代码”切到“修改代码”
- 从“修改代码”切到“运行测试”
- 开始等待权限
- 开始等待用户回答
- 明显失败
- 长时间没有对用户可见的进展

对于“长时间没有进展”的场景，建议做启发式播报，例如：

- 20 到 40 秒没有明显阶段切换时
- 补一条简短摘要

摘要内容建议控制为：

- 已完成什么
- 当前卡在哪里
- 接下来准备做什么

例如：

- 已检查登录相关代码，正在定位 token 刷新逻辑
- 已修改 2 个文件，正在跑测试
- 当前在等待权限确认，确认后会继续执行

### 中间播报的形态建议

建议组合使用：

- 更新当前任务状态卡
- 必要时补发一条简短进度消息
- 任务完成后再用完成卡收敛

优先级建议如下：

- 轻微进展：更新状态卡
- 关键阶段变化：更新状态卡 + 简短进度消息
- 需要用户决策：切换成等待权限卡 / 等待回答卡
- 已完成：发送完成卡

### 中间播报不要做成什么样

不建议：

- 每一个工具调用都推送给用户
- 每个 token 增量都更新卡片
- 把完整思考过程原文直接外露
- 把长日志直接摊平到首屏

更合适的做法是：

- 对过程做摘要
- 对阶段做归类
- 对用户能干预的节点重点提醒

## 可打断性应成为一等能力

如果希望用户真正把飞书单聊当成远程工作入口，就不能让“打断”只停留在命令能力上。

建议把打断做成显式的产品能力：

- 执行中卡首屏始终提供 `停止执行`
- 长时间运行任务在中间播报里也给出停止入口
- 如果用户发出新的明确指令，可以提示：
  - 停止当前任务并切换
  - 继续当前任务

这样用户不需要记住 `/stop`，也不需要先判断当前是不是还能中断。

## 折叠日志区的推荐内容

如果后续要做“折叠栏”，建议把它定义成“过程摘要区”，而不是“原始日志区”。

推荐内容：

- 最近一步：正在做什么
- 最近几步：已经做了什么
- 当前阻塞：为什么还没结束
- 接下来：准备做什么

不推荐内容：

- 大段逐字输出
- 未加工的内部思考
- 过长的工具原始参数

一句话原则：

- 给用户“可理解的过程”
- 不给用户“难以消费的流水账”

## 推荐的整体交互流

理想的单聊路径建议是这样：

### 路径一：首次进入

1. 用户打开机器人单聊
2. 看到首页卡
3. 点“新任务”
4. 输入自然语言任务
5. 进入任务状态卡

### 路径二：处理中

1. 用户提交任务
2. 看到“执行中卡”
3. 中途如果阶段发生变化，状态卡更新摘要
4. 如果较长时间没有明显进展，补发一条简短进度播报
5. 若需要权限，卡片切换为“等待权限卡”
6. 用户点击允许或拒绝
7. 卡片继续回到“执行中”

### 路径三：任务完成

1. 卡片切换为“完成卡”
2. 同时在消息下方挂 2 到 3 个跟随气泡
3. 用户点击“解释一下”或“运行测试”
4. 触发下一轮任务

### 路径四：继续历史工作

1. 用户从菜单点“继续上次”
2. 看到最近会话卡
3. 每张会话项展示：
   - 项目目录
   - 上次任务摘要
   - 最后活跃时间
4. 用户点选继续
5. 直接进入该任务状态卡

## 建议优先实现的交互件

如果只做最有收益的一版，建议先做这 5 个：

1. 机器人菜单
2. 首页卡
3. 任务状态卡
4. 权限决策卡
5. 完成卡 + 跟随气泡下一步

这 5 个组合起来，就能把飞书单聊从“命令入口”升级成“远程工作入口”。

## 文案建议

为了降低使用门槛，文案尽量遵循这些原则：

- 多用动作词，少用技术词
- 多说“下一步做什么”，少说“当前能力是什么”
- 多给结论，少让用户判断
- 尽量避免让用户记住命令

例如：

- 不建议：`/sessions`
- 更建议：`继续上次任务`

- 不建议：`/perm on`
- 更建议：`开启权限确认`

- 不建议：`/verbose 1`
- 更建议：`显示详细过程`

## 与命令的关系

命令不需要删除，但建议角色后移。

建议定位：

- 菜单、卡片、跟随气泡：默认入口
- 自然语言：主要输入方式
- slash 命令：高级用户和兜底入口

这样既不牺牲灵活性，也能显著降低新用户门槛。

## 一句话结论

飞书单聊更好的方向不是“继续丰富命令”，而是：

`菜单做全局入口，卡片做状态和决策，跟随气泡做下一步，自然语言做复杂表达`

只要把这四层配合起来，用户就不需要反复输入 `/sessions` 这类命令，也更容易把 tlive 当成日常工作入口来使用。
